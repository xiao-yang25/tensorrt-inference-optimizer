# 社招简历项目说明：TensorRT Inference Optimizer（BEVDet）

## 一、项目定位（写给社招面试官）

面向自动驾驶感知部署场景，构建 BEVDet 的工业化 TensorRT 推理优化与量化部署框架，覆盖：

- ONNX 到 TensorRT 引擎构建（FP16/INT8）
- 自定义 CUDA Plugin（BEVPool/Align/Gather）开发与集成
- 两阶段/单图串联推理链路打通
- 一键编译测试与可追溯验证报告
- 量化评估闭环（性能 + 数值误差）

## 二、核心贡献（简历可直接使用，3 点以上）

1. **搭建分层可维护的推理架构，降低工程复杂度**
   - 将系统拆分为：引擎构建层、运行时层、Pipeline 编排层、Plugin 层、验证与报告层。
   - 在 CMake 中抽象公共逻辑（包含 include/test 注册），减少重复配置，提高团队协作效率和可维护性。

2. **完成 BEV 核心算子 Plugin 化并建立全链路验证体系**
   - 实现 `BevPool`、`AlignBev`、`GatherBev` 的 CUDA Kernel + TensorRT Plugin 封装。
   - 建立分层验证链路：Kernel 单测 -> Plugin enqueue 冒烟 -> TRT 图内集成（含序列化/反序列化），显著降低线上回归风险。

3. **落地 INT8 PTQ 基线评估闭环，支持部署决策**
   - 构建 FP16 vs INT8 对比工具链，输出时延/吞吐与 MAE/RMSE/MaxAbs 等误差指标。
   - 增加一键报告脚本，形成可复现的量化评估流程，支持“性能收益 vs 精度损失”的工程化权衡。

4. **建设一键编译测试流程，提升交付质量**
   - 新增一键脚本串联 Configure/Build/CTest 并自动生成验证文档。
   - 产出 `reports/verification_report.md` 与日志文件，支持交接、审计与 CI 接入。

## 三、详细逻辑（面试讲述模板）

### 1）为什么采用“先两阶段打通，再插件化增强”的策略

- BEVDet 在部署中天然存在中间几何/特征构造环节，直接端到端单图改造风险高。
- 实施路径：
  - 先确保两阶段链路可运行、可计时、可定位瓶颈；
  - 再将关键中间算子迁移为 Plugin，逐步提高融合度；
  - 最后在 TRT 图内完成串联验证，保证性能与稳定性。
- 该策略兼顾了交付速度与技术风险控制。

### 2）如何保证“优化不破坏正确性”

- 推理优化（精度切换、图融合、插件替换）最常见问题是“静默精度回退”。
- 项目中采用分层防线：
  - Kernel 数值对齐（最底层正确性）
  - Plugin 接口级 `enqueue` 验证（接口与数据布局正确）
  - TRT 集成验证（序列化/反序列化 + 执行链路正确）
  - 双引擎输出误差统计（业务可感知指标）
- 能把问题快速定位到“算子实现/接口封装/图集成/量化策略”中的具体层级。

### 3）如何把量化做成“可上线”的流程

- 仅有 INT8 引擎并不等于可上线，关键在于可复现与可比较。
- 项目将量化流程标准化为：
  - 固定脚本入口与参数约束
  - 校准缓存复用机制
  - 同输入双引擎对比
  - 结构化报告沉淀（json + markdown）
- 这样可以稳定支持版本迭代中的回归分析和发布门禁。
